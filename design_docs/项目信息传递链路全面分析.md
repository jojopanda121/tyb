# 项目信息传递链路全面分析

本文档不是实施计划，而是对整个 Flow → Crew → Registry → MD → PDF 信息传递链路的诊断分析报告。

---

## 一、Flow 执行拓扑

```
@start() prepare_evidence
    │  PDF解析 → 元数据/页索引 → registry初始化 → 目录创建
    │  输出: {company_name, industry}
    ▼
@listen(prepare_evidence) build_research_plan
    │  PlanningCrew (4 tasks: scope → question_tree → registry_judgments → evidence_map)
    │  输出: 3个md文件 + RegistrySeedPlan写入registry
    ▼
@router(build_research_plan) run_research_crew
    │  顺序执行7个research sub-crew
    │  每个crew: search_facts → extract_file_facts → check_registry → synthesize_and_output
    │  输出: 7个 *_pack.md + registry更新
    ▼
@listen(RESEARCH_STAGE_COMPLETED) review_research_gate
    │  QACrew coverage审查
    │  输出: GateReviewOutput → apply_gate_review写回registry
    ▼
@router(review_research_gate) route_research_gate
    ├─ pass/force_pass ──────────────────────────────┐
    └─ retry → rerun_research (定向重跑affected_packs) │
               → 回到 review_research_gate            │
                                                      ▼
@router(or_("research_gate_passed", FORCE_PASS)) run_valuation_crew
    │  ValuationCrew (3 tasks: peers → intrinsic_value → synthesize)
    │  输出: 3个md (peers_pack, intrinsic_value_pack, valuation_pack)
    ▼
@router(VALUATION_STAGE_COMPLETED_NO_GATE) run_investment_thesis_crew
    │  InvestmentThesisCrew (2 tasks: thesis → diligence_questions)
    │  输出: 2个md
    ▼
@listen(THESIS_STAGE_COMPLETED_NO_GATE) publish_if_passed
    │  WriteupCrew (2 tasks: compile_report → export_final_report)
    │  输出: 最终 .md + .pdf
    ▼
    END
```

---

## 二、每个 Crew 的信息输入/输出明细

### 1. PlanningCrew
| 项目 | 内容 |
|------|------|
| **输入来源** | `_planning_inputs()`: company_name, industry, pdf路径, page_index, document_profile_summary |
| **Task 0** | define_research_scope → `01_research_scope.md` |
| **Task 1** | seed_question_tree → `02_question_tree.md` (context: task 0) |
| **Task 2** | seed_registry_judgments → JSON `RegistrySeedPlan` (context: task 1) |
| **Task 3** | seed_evidence_map → `03_evidence_map_seed.md` (context: task 1 + task 2) |
| **Registry写入** | Flow用 `result.tasks_output[2]` 提取seed → `replace_registry_entries()` 替换全部entries |
| **State更新** | research_scope_path, question_tree_path, evidence_map_seed_path |

### 2. 七个Research Sub-Crews (统一模式)
| 项目 | 内容 |
|------|------|
| **公共输入** | `_base_inputs()`: company_name, industry, pdf路径, question_tree_text, research_scope_text |
| **特有输入** | pack_name, pack_title, pack_output_path, loop_reason, qa_feedback + 各crew特定guidance |
| **上游依赖** | peer_info_crew ← industry_pack_text + business_pack_text; finance_crew/operating_metrics_crew ← peer_info_pack_text |
| **4个Agent** | search_fact_agent, extract_file_fact_agent, qa_check_agent, synthesizing_agent |
| **4个Task** | search_facts → extract_file_facts → check_registry → synthesize_and_output |
| **Registry交互** | 每个agent通过tools: AddEntryTool, AddEvidenceTool, StatusUpdateTool, RegistryReviewTool |
| **输出** | `XX_*_pack.md` (via task output_file) |
| **Flow后处理** | `_register_pack_output()`: 将pack文本前800字符作为context evidence注册到registry judgment entries |

**7个crew执行顺序及输出:**
1. HistoryBackgroundCrew → `01_history_background_pack.md`
2. IndustryCrew → `02_industry_pack.md`
3. BusinessCrew → `03_business_pack.md`
4. PeerInfoCrew → `04_peer_info_pack.md` (依赖: industry + business)
5. FinancialCrew → `05_finance_pack.md` (依赖: peer_info)
6. OperatingMetricsCrew → `06_operating_metrics_pack.md` (依赖: peer_info)
7. RiskCrew → `07_risk_pack.md`

### 3. QACrew (Research Gate)
| 项目 | 内容 |
|------|------|
| **输入** | `_base_inputs()` + stage_bundle(7个pack拼接) + registry_markdown + registry_diff + registry_full_text |
| **Task** | review_question_coverage → `GateReviewOutput` (JSON) |
| **Registry写入** | Flow调用 `apply_gate_review()`: 将affected_packs对应的judgment entries状态更新为supported/gap/deferred |
| **路由决策** | pass→估值, revise→定向重跑(max 1次), 超限→force_pass→估值 |

### 4. ValuationCrew
| 项目 | 内容 |
|------|------|
| **输入** | `_base_inputs()` + peer_info_pack_text + finance_pack_text + operating_metrics_pack_text + risk_pack_text |
| **3个Task** | analyze_peers → calculate_intrinsic_value → synthesize_valuation |
| **输出** | `01_peers_pack.md`, `02_intrinsic_value_pack.md`, `03_valuation_pack.md` |
| **Registry写入** | peers_pack和intrinsic_value_pack通过 `_register_pack_output()` 注册为context evidence |

### 5. InvestmentThesisCrew
| 项目 | 内容 |
|------|------|
| **输入** | `_base_inputs()` + 全部7个research pack文本 + peers_pack_text + valuation_pack_text + registry_full_text |
| **2个Task** | synthesize_investment_thesis → design_diligence_questions |
| **特点** | reasoning=True (唯一开启推理的crew) |
| **输出** | `01_investment_thesis.md`, `02_diligence_questions.md` |

### 6. WriteupCrew
| 项目 | 内容 |
|------|------|
| **输入** | `_base_inputs()` + 全部research/valuation/thesis文本 + final_qa_summary + 最终路径 |
| **Task 1** | compile_report → 最终markdown (output_file指向final_report_markdown_path) |
| **Task 2** | export_final_report → 调用MarkdownToPdfTool生成PDF (context依赖task 1) |
| **特点** | report_editor无任何tools，纯文本拼接; temperature=0.1 |

## 三、Registry 信息流转全景

```
prepare_evidence
  └→ initialize_registry(): 10条默认seed entries (company_name, industry)
       ▼
build_research_plan
  └→ replace_registry_entries(RegistrySeedPlan): 替换为planner生成的30-50条entries
     (entry_origin="planner", 状态="open")
       ▼
7个research crews (每个crew内部):
  ├→ add_discovered_entry(): 新增fact/data/judgment (origin="discovered")
  ├→ register_evidence(): 关联evidence到entries (更新stance→支持/冲突/上下文)
  ├→ update_entry_status(): 将entries标记为supported/confirmed/gap
  └→ record_registry_review(): 写入audit trail notes
       ▼
Flow._register_pack_output() (每个crew完成后):
  └→ register_evidence(stance="context"): 将pack输出前800字符注册为context evidence
       ▼
review_research_gate:
  └→ apply_gate_review(): 按affected_packs更新judgment entries (pass→supported, revise→gap)
       ▼
valuation/thesis crews:
  └→ 同样通过tools读写registry
       ▼
publish_if_passed:
  └→ WriteupCrew只读registry (通过flow传入的registry文本)，不再写入
```

**Registry 读取方式:**
- Agent通过 `ReadRegistryTool`: view="markdown" (表格视图) / view="full" (完整JSON) / view="entry_detail" / view="evidence_detail"
- Flow通过 `render_registry_markdown()`, `entry_ids_for_packs()`, `build_registry_diff_summary()`
- Writeup通过flow传入的 `registry_full_text` (未在task模板中显式引用，但在base_inputs中)

## 四、MD → PDF 生成链路

1. **compile_report task**: 按固定10章模板拼接所有pack文本 → 写入 `{final_report_markdown_path}`
2. **export_final_report task**: 调用 `MarkdownToPdfTool` → 读取md → 正则化表格 → markdown→HTML(含extensions) → BeautifulSoup处理表格CSS → WeasyPrint生成A4 PDF (含中文字体支持)

**最终报告章节结构:**
```
# {company_name} 研究报告
## 1. 投资逻辑          ← investment_thesis_text
## 2. 尽调问题          ← diligence_questions_text
## 3. 公司历史及股东     ← history_background_pack_text
## 4. 行业分析          ← industry_pack_text
## 5. 业务分析          ← business_pack_text + operating_metrics_pack_text
## 6. 财务分析          ← finance_pack_text
## 7. 风险分析          ← risk_pack_text
## 8. 可比公司情况       ← peer_info_pack_text
## 9. 综合估值          ← valuation_pack_text
## 10. QA摘要与结论边界  ← final_qa_summary
```

## 五、发现的链路问题与风险点

### P1: 关键数据断裂

**[问题1] Valuation子产物在最终报告中丢失**
- **位置**: [tasks.yaml:53](src/automated_research_report_generator/crews/writeup_crew/config/tasks.yaml#L53) (section 9)
- **现象**: Flow在 `publish_if_passed()` 中向writeup传入了 `peers_pack_text` 和 `intrinsic_value_pack_text`，但 compile_report task模板中 **从未引用** 这两个变量
- **结果**: Valuation阶段的 `01_peers_pack.md`(可比公司估值倍数分析) 和 `02_intrinsic_value_pack.md`(内在价值分析) 的详细内容完全未进入最终报告
- **影响**: Section 9 只展示 `valuation_pack_text` (合成后的估值概要)，估值推导过程的细节丢失
- **严重度**: 中高 — 如果 valuation_pack 是完整合成则影响可控，但如果只是摘要则关键估值推导链断裂

**[问题2] Thesis阶段未接收 intrinsic_value_pack_text**
- **位置**: [research_flow.py:420-433](src/automated_research_report_generator/flow/research_flow.py#L420-L433) `_run_thesis_stage()`
- **现象**: Thesis crew收到 peers_pack_text 和 valuation_pack_text，但 **没有** intrinsic_value_pack_text
- **结果**: 投资论点合成时无法直接引用 DCF/内在价值模型的详细假设
- **影响**: 如果 valuation_pack 的合成已经充分包含intrinsic value细节则无影响，否则thesis判断缺乏估值根据

### P2: 结构性脆弱点

**[问题3] Registry seed提取使用硬编码索引**
- **位置**: [research_flow.py:261](src/automated_research_report_generator/flow/research_flow.py#L261)
- **现象**: `result.tasks_output[2]` 硬编码假设 `seed_registry_judgments` 是第3个task
- **风险**: PlanningCrew现在有4个task (scope→tree→**judgments**→map)，索引[2]正确。但如果未来调整task顺序或增删task，会静默取错task产物
- **建议**: 改为按task名称查找，或在PlanningCrew中显式标记seed输出

**[问题4] Consistency review被禁用但路由逻辑仍检查**
- **位置**: [research_flow.py:341](src/automated_research_report_generator/flow/research_flow.py#L341) `run_consistency_review=False`
- **现象**: `_route_gate()` 检查 `[coverage.status, consistency.status]` 是否全部pass，但consistency永远返回一个 "deferred" placeholder (status="pass")
- **影响**: 当前不影响正确性（因为placeholder status="pass"），但如果未来placeholder改为非pass状态，会意外阻断流程

### P3: 信息冗余/冗余传递

**[问题5] QA阶段接收空的thesis/diligence输入**
- **位置**: [research_flow.py:933-934](src/automated_research_report_generator/flow/research_flow.py#L933-L934)
- **现象**: Research gate阶段的 `_run_qa_stage()` 传入 `investment_thesis_text` 和 `diligence_questions_text`，但此时thesis尚未生成，均为空字符串
- **影响**: 无功能bug，但增加了prompt token消耗和理解成本

**[问题6] history_governance_pack_text 传入writeup但未使用**
- **位置**: [research_flow.py:483](src/automated_research_report_generator/flow/research_flow.py#L483)
- **现象**: 这是 history_background_pack 的legacy别名，指向同一个文件。传入writeup但task模板未引用
- **影响**: 无数据丢失（内容与history_background_pack_text相同），但增加legacy复杂度

### P4: 其他观察

**[观察1] Valuation阶段只接收4/7个research pack**
- **位置**: [research_flow.py:859-867](src/automated_research_report_generator/flow/research_flow.py#L859-L867)
- **传入**: peer_info + finance + operating_metrics + risk
- **未传入**: history_background, industry, business
- **判断**: 可能是有意设计（估值聚焦财务数据），但industry_pack中的行业增长率/竞争格局对估值假设有潜在参考价值

**[观察2] _register_pack_output 摘要截断800字符**
- **位置**: [research_flow.py:1094](src/automated_research_report_generator/flow/research_flow.py#L1094)
- **影响**: Registry中pack的context evidence只保留前800字符，后续QA基于registry审查时可能漏掉pack后半部分的发现
- **缓解**: QA同时也接收完整pack文本作为stage_bundle

**[观察3] final_qa_summary 只包含research QA结果**
- **位置**: [research_flow.py:1119-1137](src/automated_research_report_generator/flow/research_flow.py#L1119-L1137)
- **影响**: 报告最终章节"QA摘要与结论边界"只展示研究阶段覆盖度/一致性状态，不包含估值和thesis阶段的质检信息（因为这两个阶段已取消外部gate）

## 六、链路健康度总结

| 链路环节 | 状态 | 说明 |
|---------|------|------|
| PDF → prepare_evidence | **正常** | 元数据/页索引/registry初始化完整 |
| prepare_evidence → PlanningCrew | **正常** | 输入完整，seed写入registry正确 |
| PlanningCrew → Registry seed | **脆弱** | 硬编码索引[2]，功能正确但不鲁棒 |
| Research crew间依赖 | **正常** | peer←(industry+business), finance/ops←peer_info |
| Research crew → Registry | **正常** | 4种tool覆盖完整的CRUD操作 |
| Research → QA gate | **正常** | 7个pack文本+registry完整传入 |
| QA gate → Registry回写 | **正常** | apply_gate_review正确更新judgment状态 |
| QA gate → 定向重跑 | **正常** | affected_packs→targeted_packs映射正确 |
| Research → Valuation | **部分缺失** | 只传4/7个pack，缺industry/history/business |
| Valuation → Thesis | **轻微缺失** | 缺intrinsic_value_pack_text |
| Thesis → Writeup | **正常** | 全部文本传入 |
| Writeup task模板 | **有数据丢失** | peers_pack_text, intrinsic_value_pack_text未被模板引用 |
| compile_report → PDF | **正常** | MarkdownToPdfTool链路完整 |
| Registry全程一致性 | **正常** | 线程安全，checkpoint/snapshot机制完善 |
